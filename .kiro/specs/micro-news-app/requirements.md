# Requirements Document

## Introduction

Micro-News is a two-part web application that automates the collection, summarization, and delivery of short news snippets to subscribed users. The Admin Application allows administrators to manage news categories and monitor the system. The User Application allows users to register, manage their profile, subscribe to news categories, and receive daily personalized news digests.

The system uses a CrewAI agent framework to autonomously search the internet for news based on configured categories, summarize articles into micro-snippets (≤60 words), and make them available in each subscriber's in-app web feed at their preferred time each day. Snippets are delivered exclusively via the in-app web feed; no email delivery of Snippets is provided.

The first Admin account is bootstrapped via environment variables at application startup; no setup page or seed script is used for initial admin creation.

## Glossary

- **Admin_App**: The web application used by administrators to manage categories and monitor the system.
- **User_App**: The user-facing web application for registration, profile management, subscriptions, and news reading.
- **Admin**: An authenticated administrator who manages the Micro-News system.
- **User**: A registered and authenticated end-user of the User_App.
- **Category**: A named news topic (e.g., "Technology", "Sports") used to group and filter news.
- **Snippet**: A news summary of no more than 60 words generated by the News_Agent from a source article.
- **News_Agent**: The CrewAI-based agent responsible for searching the internet and generating Snippets.
- **Scheduler**: The system component that triggers the News_Agent at 09:00 IST (India Standard Time, UTC+5:30) every day, regardless of the server's local timezone.
- **Delivery_Service**: The system component that makes Snippets available in the User's in-app web feed at their preferred delivery time.
- **Subscription**: A relationship between a User and a Category indicating the User wants to receive Snippets for that Category.
- **Preferred_Delivery_Time**: A time-of-day value set by the User in their profile, at which the Delivery_Service sends daily Snippets.
- **Email_Validator**: The component that verifies an email address format and existence during registration.
- **Password_Reset_Service**: The component that handles password change requests via registered email.

## Requirements

### Requirement 1: Admin Authentication

**User Story:** As an Admin, I want to log in with a secured username and password, so that only authorized personnel can access the Admin_App.

#### Acceptance Criteria

1. WHEN an Admin submits valid credentials, THE Admin_App SHALL authenticate the Admin and grant access to the admin dashboard.
2. WHEN an Admin submits invalid credentials, THE Admin_App SHALL reject the login attempt and display an error message without revealing which field is incorrect.
3. WHILE an Admin session is active, THE Admin_App SHALL protect all admin routes from unauthenticated access.
4. WHEN an Admin session exceeds 60 minutes of inactivity, THE Admin_App SHALL invalidate the session and redirect the Admin to the login page.
5. IF more than 5 consecutive failed login attempts occur from the same source within 10 minutes, THEN THE Admin_App SHALL temporarily block further login attempts from that source for 15 minutes.

---

### Requirement 2: Category Management

**User Story:** As an Admin, I want to create and delete news categories, so that I can control which topics the system collects news for.

#### Acceptance Criteria

1. WHEN an Admin submits a new category name, THE Admin_App SHALL create the Category and persist it to the database.
2. WHEN an Admin submits a category name that already exists (case-insensitive), THE Admin_App SHALL reject the creation and display a duplicate-name error.
3. WHEN an Admin deletes a Category, THE Admin_App SHALL remove the Category and all associated Snippets from the system.
4. WHEN an Admin deletes a Category that has active Subscriptions, THE Admin_App SHALL notify the Admin of the number of affected Users before confirming deletion.
5. THE Admin_App SHALL display all existing Categories in a paginated list sorted alphabetically.
6. THE Admin_App SHALL NOT provide a category edit or rename function.

---

### Requirement 3: Automated News Collection

**User Story:** As an Admin, I want the system to automatically collect and summarize news every morning, so that fresh Snippets are available for delivery to Users.

#### Acceptance Criteria

1. WHEN the Scheduler triggers at 09:00 IST (UTC+5:30) each day, THE News_Agent SHALL search the internet for recent news articles for each active Category.
2. WHEN the News_Agent retrieves a news article, THE News_Agent SHALL generate a Snippet of no more than 60 words summarizing the article.
3. WHEN the News_Agent generates a Snippet, THE News_Agent SHALL associate the Snippet with the corresponding Category and persist it to the database.
4. IF the News_Agent fails to retrieve articles for a Category, THEN THE News_Agent SHALL log the failure with the Category name, timestamp, and error reason, and continue processing remaining Categories.
5. WHEN a news collection run completes, THE Scheduler SHALL record the run timestamp, total Snippets generated, and any per-Category failures.
6. THE News_Agent SHALL generate at most 10 Snippets per Category per daily collection run.

---

### Requirement 4: User Registration

**User Story:** As a visitor, I want to register with my email address and a password, so that I can access the Micro-News service.

#### Acceptance Criteria

1. WHEN a visitor submits a registration form with a valid email and password, THE User_App SHALL create a new User account and send a verification email to the provided address.
2. WHEN a visitor submits a registration form with an email that does not conform to RFC 5322 format, THE Email_Validator SHALL reject the submission and display a format error.
3. WHEN a visitor submits a registration form with an email address already registered, THE User_App SHALL reject the submission and display an account-exists error.
4. WHEN a visitor submits a password shorter than 8 characters or lacking at least one uppercase letter, one lowercase letter, and one digit, THE User_App SHALL reject the submission and display the specific password policy violation.
5. WHEN a visitor clicks the verification link in the email, THE User_App SHALL activate the User account and allow login.
6. IF a verification link is not clicked within 24 hours, THEN THE User_App SHALL expire the link and require the visitor to request a new verification email.

---

### Requirement 5: User Authentication

**User Story:** As a User, I want to log in securely, so that I can access my personalized news feed and profile.

#### Acceptance Criteria

1. WHEN a User submits valid credentials, THE User_App SHALL authenticate the User and grant access to the user dashboard.
2. WHEN a User submits invalid credentials, THE User_App SHALL reject the login attempt and display a generic error message without revealing which field is incorrect.
3. WHILE a User session is active, THE User_App SHALL protect all authenticated routes from unauthenticated access.
4. WHEN a User session exceeds 120 minutes of inactivity, THE User_App SHALL invalidate the session and redirect the User to the login page.

---

### Requirement 6: User Profile Management

**User Story:** As a User, I want to manage my profile, so that I can personalize my experience and control how I receive news.

#### Acceptance Criteria

1. WHEN a logged-in User updates their profile name, THE User_App SHALL persist the new name and display it across the application.
2. WHEN a logged-in User updates their birthday, THE User_App SHALL validate that the date is a valid calendar date in the past and persist it.
3. WHEN a logged-in User sets a Preferred_Delivery_Time, THE User_App SHALL persist the time value and use it to schedule daily Snippet delivery.
4. THE User_App SHALL display the User's current profile name, birthday, and Preferred_Delivery_Time on the profile page.
5. WHEN a logged-in User submits a change-password request, THE Password_Reset_Service SHALL send a password-reset link to the User's registered email address.
6. WHEN a User clicks a valid password-reset link and submits a new password meeting the password policy, THE Password_Reset_Service SHALL update the User's password and invalidate the reset link.
7. IF a password-reset link is not used within 1 hour, THEN THE Password_Reset_Service SHALL expire the link.

---

### Requirement 7: News Subscription Management

**User Story:** As a User, I want to subscribe and unsubscribe to news Categories, so that I receive only the topics I care about.

#### Acceptance Criteria

1. WHEN a logged-in User subscribes to a Category, THE User_App SHALL create a Subscription linking the User to the Category.
2. WHEN a logged-in User unsubscribes from a Category, THE User_App SHALL remove the Subscription and stop delivering Snippets for that Category to the User.
3. THE User_App SHALL display all available Categories with a clear indication of which ones the User is currently subscribed to.
4. WHEN a logged-in User has no active Subscriptions, THE User_App SHALL display a prompt encouraging the User to subscribe to at least one Category.

---

### Requirement 8: Daily News Delivery

**User Story:** As a User, I want to receive daily news Snippets at my preferred time, so that I can stay informed without information overload.

#### Acceptance Criteria

1. WHEN the Delivery_Service reaches a User's Preferred_Delivery_Time, THE Delivery_Service SHALL make all undelivered Snippets from the current day's collection for the User's subscribed Categories available in the User's in-app web feed.
2. WHEN a User has not set a Preferred_Delivery_Time, THE Delivery_Service SHALL default to making Snippets available in the User's web feed at 10:00 IST (UTC+5:30).
3. IF no new Snippets are available for a User's subscribed Categories on a given day, THEN THE Delivery_Service SHALL NOT update the User's web feed with a delivery notification.
4. WHEN the Delivery_Service publishes Snippets to a User's web feed, THE Delivery_Service SHALL record the delivery timestamp for each Snippet.

---

### Requirement 9: News Feed Interaction

**User Story:** As a User, I want to mark news as read and delete news messages, so that I can manage my personal news feed.

#### Acceptance Criteria

1. WHEN a logged-in User marks a Snippet as read, THE User_App SHALL update the Snippet's status to "read" for that User and visually distinguish it from unread Snippets.
2. WHEN a logged-in User deletes a Snippet from their feed, THE User_App SHALL remove the Snippet from the User's feed view without affecting other Users' feeds.
3. THE User_App SHALL display unread Snippets before read Snippets within each Category section of the feed.
4. THE User_App SHALL display the total count of unread Snippets on the user dashboard.

---

### Requirement 10: Admin Account Bootstrapping

**User Story:** As a system operator, I want the first Admin account to be created from environment variables at startup, so that no manual setup step or seed script is required to initialize the system.

#### Acceptance Criteria

1. WHEN the application starts and no Admin account exists in the database, THE Admin_App SHALL create an Admin account using the username and password values read from the designated environment variables.
2. WHEN the required environment variables for admin bootstrapping are absent at startup and no Admin account exists, THE Admin_App SHALL log a descriptive error and halt startup.
3. WHEN an Admin account already exists in the database at startup, THE Admin_App SHALL skip the bootstrapping step regardless of the environment variable values.
4. THE Admin_App SHALL NOT expose a web-based setup page or rely on a seed script for initial Admin account creation.

---

### Requirement 11: Responsive Web Design

**User Story:** As a User, I want the User_App to be usable on any mobile device, so that I can read my news on the go.

#### Acceptance Criteria

1. THE User_App SHALL render all pages using a responsive layout that adapts to viewport widths from 320px to 2560px.
2. WHEN a User accesses the User_App on a viewport narrower than 768px, THE User_App SHALL display a mobile-optimized navigation menu.
3. THE User_App SHALL ensure all interactive elements (buttons, links, form inputs) have a minimum touch target size of 44x44 CSS pixels on mobile viewports.
